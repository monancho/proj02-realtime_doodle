# 요구사항 정의서

## 서비스 개요

Realtime Doodle Relay는 웹 기반 실시간 협업 낙서 미니 프로젝트다. 사용자는 로그인 후 방을 만들거나 방 코드로 입장하고, 각자 사진을 업로드한다. 라운드가 시작되면 서버가 미사용 사진 중 하나를 무작위로 선택하고, 참가자들은 제한 시간 동안 같은 사진 위에 그림을 그리며 채팅한다. 라운드 종료 후 결과 이미지를 저장하고 최종 갤러리에서 확인/다운로드한다.

## 사용자 유형

| 사용자 | 설명 | 주요 행동 |
|---|---|---|
| 비로그인 사용자 | 서비스 접근 전 사용자 | 회원가입, 로그인 |
| 일반 사용자 | 방을 만들거나 참여하는 사용자 | 사진 업로드, 채팅, 드로잉, 결과 다운로드 |
| 방장 | 방을 생성한 사용자 | 게임 시작, 라운드 진행 제어 |

## MVP 포함 범위

- Firebase Authentication 기반 회원가입/로그인
- 방 생성과 6자리 방 코드 기반 입장
- 사용자별 사진 업로드
- 업로드 이미지 MIME/용량 검증
- MongoDB GridFS 기반 원본 이미지 저장
- 라운드마다 미사용 이미지 랜덤 선택
- Canvas 기반 펜/지우개/색상/굵기 도구
- Socket.IO 기반 stroke 동기화
- 같은 방 사용자끼리 실시간 채팅
- 서버 기준 라운드 타이머
- 라운드 종료 후 결과 이미지 저장
- 결과 갤러리 조회와 다운로드
- Render, MongoDB Atlas, Firebase Auth 기반 MVP 배포

## MVP 제외 범위

- SNS 공유
- 랭킹/투표
- 결제
- 관리자 페이지
- 모바일 앱
- 모바일 완전 최적화
- Redis 기반 Socket.IO 수평 확장
- Undo/Redo
- 고급 이미지 편집
- 영구 채팅 아카이브
- 복잡한 권한 시스템

## 기능 요구사항

| ID | 요구사항 | 우선순위 |
|---|---|---|
| FR-001 | 사용자는 Firebase Authentication을 통해 회원가입과 로그인을 할 수 있어야 한다. | 상 |
| FR-002 | 로그인한 사용자는 방을 생성하고 방 코드를 공유할 수 있어야 한다. | 상 |
| FR-003 | 사용자는 방 코드로 기존 방에 입장할 수 있어야 한다. | 상 |
| FR-004 | 방 참여자는 사진 파일을 업로드할 수 있어야 한다. | 상 |
| FR-005 | 서버는 업로드된 사진 중 하나를 라운드마다 랜덤으로 선택해야 한다. | 상 |
| FR-006 | 사용자는 선택된 사진 위 Canvas에 그림을 그릴 수 있어야 한다. | 상 |
| FR-007 | 드로잉 좌표는 같은 방의 다른 사용자에게 실시간 반영되어야 한다. | 상 |
| FR-008 | 사용자는 같은 방 사용자와 실시간 채팅할 수 있어야 한다. | 상 |
| FR-009 | 라운드 타이머가 종료되면 최종 결과 이미지가 저장되어야 한다. | 상 |
| FR-010 | 게임 종료 후 사용자는 결과 갤러리에서 모든 결과물을 볼 수 있어야 한다. | 상 |
| FR-011 | 사용자는 원하는 결과 이미지를 다운로드할 수 있어야 한다. | 중 |
| FR-012 | 사용자는 참가자 목록과 현재 라운드 상태를 확인할 수 있어야 한다. | 중 |

## 비기능 요구사항

| 구분 | 요구사항 |
|---|---|
| 성능 | 드로잉 이벤트는 stroke 단위 또는 throttle 방식으로 전송한다. |
| 보안 | API와 Socket 연결은 Firebase ID Token 검증 후 허용한다. |
| 안정성 | Render 로컬 파일 시스템에 영구 저장하지 않고 MongoDB Atlas와 GridFS를 사용한다. |
| 호환성 | 최소 Chrome 최신 버전 기준으로 동작한다. |
| 확장성 | MVP는 단일 Render 인스턴스 기준으로 구성하고 다중 인스턴스 확장은 후순위로 둔다. |
| 사용성 | 드로잉 툴은 펜 색상, 굵기, 지우개 정도로 단순화한다. |
